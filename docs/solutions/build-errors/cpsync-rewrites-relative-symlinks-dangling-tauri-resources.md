# fs.cpSync rewrites relative symlinks to absolute paths — vendored npm trees dangle and break tauri build

## Problem

`npm run release` aborted during Tauri bundling because two npm `.bin` symlinks inside the vendored Steel tree (`src-tauri/resources/steel/`) pointed at absolute paths in a temporary build directory that had already been deleted, and the Tauri bundler refuses to package a resource path that cannot be stat'ed.

## Symptoms

The release script (`package.json:27` — `"release": "npm run build:sidecar && npm run tauri:build"`) completed the sidecar phase, including the Steel vendoring step (`scripts/build-sidecar.ts:271` calls `buildSteelBundle()`), then died inside `tauri build` with:

```
resource path `resources/steel/node_modules/archiver-utils/node_modules/.bin/glob` doesn't exist
failed to build app: failed to build app
```

Two properties of the failure were the tells:

- The same error also fired for a second entry, `resources/steel/node_modules/fastify/node_modules/.bin/pino` — both nested `.bin` links, not ordinary files.
- `ls -l` on the vendored tree showed both symlinks resolving into `/private/var/folders/.../T/comate-steel-build-fAut73/steel-browser/...` — an absolute path inside a `mkdtempSync` work dir (`scripts/build-steel-bundle.ts:448` creates `$TMPDIR/comate-steel-build-XXXXXX`) that the build's `finally` block had already removed (`scripts/build-steel-bundle.ts:485-487`). Exactly two symlinks existed in the entire vendored tree and both dangled.

The failure is deterministic on a fresh Steel build but invisible otherwise: once a broken tree is in place, every `tauri build` fails the same way until the tree is replaced.

## What Didn't Work

- **Deleting the two dangling symlinks (or the nested `.bin` directories).** This unblocks exactly one `tauri build`, but the fix is cosmetic: nothing is wrong with the links themselves — npm created them correctly as *relative* links (`../glob/dist/esm/bin.mjs`, `../pino/bin.js`). They became absolute-into-temp during the copy step, so the next `npm run build:steel` re-vendors the tree and recreates the same dangling links. The breakage is in the copy, not in npm's output.
- **Recreating the temp dir so the absolute targets resolve.** Absurd on its face, but worth naming because it is the only "fix" consistent with treating the symlink targets as correct: the temp path embeds a random suffix (`comate-steel-build-fAut73`), is deleted in a `finally` on every run, and would couple the shipped app bundle to a scratch directory on one build machine.
- **Running `tauri build` without rebuilding Steel.** Works only as long as `src-tauri/resources/steel/` is left untouched, and it masks the bug rather than fixing it — the vendored tree is regenerated by every `npm run release` (release → build:sidecar → `buildSteelBundle()`), so the very next release reintroduces the dangling links.
- **Suspecting npm, the lockfile, or the Steel pin.** A dead end: the symlinks in the *checkout* are fine. The corruption happens in this repo's own staging/install copies, which is why an isolated Node experiment (create a relative symlink, `cpSync` it with default options vs. `verbatimSymlinks: true`, compare `readlink`) was the decisive step — it reproduced the relative→absolute rewrite without involving npm or Tauri at all.

## Solution

Two changes, both in the vendoring pipeline.

**1. Copy npm trees with `verbatimSymlinks: true`.** `scripts/build-steel-bundle.ts` now defines a single options constant with an explanatory comment (`scripts/build-steel-bundle.ts:110-116`):

```ts
// fs.cpSync's default resolves symlink targets against the SOURCE tree and
// writes absolute links at the destination. For npm trees that rewrites
// relative `.bin` links (e.g. archiver-utils/node_modules/.bin/glob) into
// absolute paths inside the temp build dir — dangling once it is deleted,
// which then fails `tauri build` ("resource path ... doesn't exist").
// verbatimSymlinks keeps the original relative links valid after the move.
const COPY_TREE_OPTS = { recursive: true, verbatimSymlinks: true } as const;
```

The three call sites that copy symlink-containing trees changed from the default to `COPY_TREE_OPTS`. The production-closure package loop in `stageVendoredTree` (`scripts/build-steel-bundle.ts:400`):

```ts
// before
cpSync(join(checkout, lockPath), join(staging, rel), { recursive: true });
// after
cpSync(join(checkout, lockPath), join(staging, rel), COPY_TREE_OPTS);
```

And both install copies in `build()` (`scripts/build-steel-bundle.ts:476` and `scripts/build-steel-bundle.ts:478`):

```ts
// before
cpSync(staging, swapDir, { recursive: true });
rmSync(destDir, { recursive: true, force: true });
cpSync(swapDir, destDir, { recursive: true });
// after
cpSync(staging, swapDir, COPY_TREE_OPTS);
rmSync(destDir, { recursive: true, force: true });
cpSync(swapDir, destDir, COPY_TREE_OPTS);
```

**2. Fail the vendoring step on any dangling symlink, before Tauri ever sees the tree.** A new audit pair was added alongside the existing native-artifact gate in `src/server/utils/native-artifact-audit.ts`. `findDanglingSymlinks` (`src/server/utils/native-artifact-audit.ts:85-102`) walks the tree, and for each symlink uses `existsSync` — which follows links — to detect a missing target; symlinked directories are deliberately not recursed into, matching `walkFiles`:

```ts
export function findDanglingSymlinks(dir: string): string[] {
  const offenders: string[] = [];
  const walk = (current: string): void => {
    for (const entry of readdirSync(current, { withFileTypes: true })) {
      const full = join(current, entry.name);
      if (entry.isSymbolicLink()) {
        // existsSync follows the link, so a dangling target reads as false.
        if (!existsSync(full)) {
          offenders.push(relative(dir, full));
        }
      } else if (entry.isDirectory()) {
        walk(full);
      }
    }
  };
  walk(dir);
  return offenders;
}
```

`assertNoDanglingSymlinks` (`src/server/utils/native-artifact-audit.ts:104-114`) throws with an offender list and a message that names both the Tauri failure mode and the fix. It runs as a "Symlink audit" gate in `build()` after the size-budget check (`scripts/build-steel-bundle.ts:467-469`):

```ts
console.log('\n--- Symlink audit ---');
assertNoDanglingSymlinks(staging);
console.log('clean (no dangling symlinks)');
```

Note the gate runs on the *staging* tree, which still lives inside the temp work dir at that point — so it catches links that are already dangling on creation (e.g. a genuinely missing target), while `verbatimSymlinks` is what prevents the copy itself from manufacturing the temp-dir-pointing kind.

## Why This Works

The root cause is a Node `fs.cpSync`/`fs.cp` default that is surprising exactly where npm trees are concerned. With `verbatimSymlinks: false` (the default), when cpSync encounters a symlink it resolves the link's target **against the source tree** and writes the destination symlink with that **resolved absolute path** as its link text. npm creates `.bin` entries as relative symlinks — Steel's pinned checkout (`STEEL_COMMIT` at `scripts/build-steel-bundle.ts:81`, installed via `npm ci ... -w api` at `scripts/build-steel-bundle.ts:339`) contains, among others:

```
node_modules/archiver-utils/node_modules/.bin/glob -> ../glob/dist/esm/bin.mjs
node_modules/fastify/node_modules/.bin/pino        -> ../pino/bin.js
```

Copying those with default options rewrote the link text to absolute paths under the *source* root, which was the temp checkout (`$TMPDIR/comate-steel-build-XXX/steel-browser/...`). The staging copy introduced the absolute targets; the two install copies (staging → `steel.next` → `resources/steel`) re-resolved and propagated the same absolute text. Then the `finally` block deleted the work dir, both links dangled, and `tauri build` — which bundles the whole `./resources` directory per `src-tauri/tauri.conf.json:43-45` — stat'ed a link whose target no longer existed and aborted with "resource path ... doesn't exist".

`verbatimSymlinks: true` disables the resolution: cpSync copies the link *text* verbatim, so `.bin/glob` keeps reading `../glob/dist/esm/bin.mjs` at the destination. That relative link is correct in the vendored tree because the closure computation vendors every production dependency as a sibling package — `glob` and `pino` are both in the production closure, so the relative targets exist at the same relative location after the move. Relative links are location-independent; absolute-into-temp links were never valid anywhere except the temp checkout.

The general rule: **any `cpSync`/`fs.cp` of an npm install tree must set `verbatimSymlinks: true` (preserve links) or `dereference: true` (copy targets as real files) — never the default.** The default silently rewrites `.bin` links to point back into the source location, which is fatal whenever the source is a temp dir, a cache that gets pruned, or a path that differs between build and package machines.

## Prevention

- **Build gate.** `assertNoDanglingSymlinks` now runs on every Steel vendoring (`scripts/build-steel-bundle.ts:467-469`), so a future dangling link — from a new copy site, an npm behavior change, or a dependency that ships its own symlinks — fails `npm run build:steel` with a message that names the offenders and the Tauri error they would cause, instead of failing `tauri build` cryptically minutes later.
- **Tests.** `src/server/utils/native-artifact-audit.test.ts` covers the gate with four symlink cases (skipped on win32 via the `symlinkIt` guard at line 142, since symlink creation needs elevated privileges there): valid relative `.bin`-shaped links pass (lines 144-158), links to existing directories pass (lines 160-168), relative links with missing targets are flagged (lines 170-183), and — the regression test for this exact bug — absolute links into a deleted build dir are flagged (lines 185-203). The regression case mirrors the failure shape directly:

```ts
symlinkSync(
  join(tmpdir(), 'comate-steel-build-gone/steel-browser/node_modules/pino/bin.js'),
  join(binDir, 'pino'),
);
const offenders = findDanglingSymlinks(dir);
assert.strictEqual(offenders.length, 1);
assert.throws(() => assertNoDanglingSymlinks(dir), /resource path/);
```

- **Checklist when copying `node_modules` trees in build scripts:**
  1. After any script copies an npm tree, assert there are no dangling symlinks *before* packaging (mirror the staging-tree gate pattern above).
  2. Choose `verbatimSymlinks: true` or `dereference: true` deliberately — never accept the default — whenever the source may contain symlinks. Elsewhere in this repo the plugin downloader already uses `dereference: true` (`src/server/utils/plugin-downloader.ts:133`, `:199`, `:353`) and the skills installer does the same (`src/server/services/skills/installer.ts:137`); both are safe because they materialize link targets as real files. Copies of trees known to contain no symlinks (the pristine Steel checkout before `npm ci`, the plugin marketplace copy at `scripts/build-sidecar.ts:263`) are unaffected.
  3. Treat Tauri's `resource path ... doesn't exist` as a symptom, not a cause: a `bundle.resources` glob over a whole directory (`src-tauri/tauri.conf.json:43-45`) surfaces *any* unstat'able entry this way, and a broken symlink is the first thing to look for. Read the link with `ls -l`/`readlink` — if the target names a temp dir, the copy step rewrote it.
