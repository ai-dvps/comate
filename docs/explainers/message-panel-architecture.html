0一句话总览

历史消息和实时消息是两条完全不同的数据通道，但它们汇入同一个 Zustand 数组 messages[sessionId]，从那里开始共享同一条渲染管线。理解这个 panel，就是理解「两条注入通道 + 一条渲染管线」。

实时通道没有独立的「流式缓冲区」：正在生成的消息就躺在 messages[sessionId] 里，用消息上的 isStreaming 和 part 上的 state: 'streaming' 做标记。「流式结束」不是一次数据迁移，只是把标记翻掉（chat-store.ts:1630-1642）。这就是为什么历史与实时可以用同一套组件渲染——它们在 store 里本来就长一个样。

1组件结构：一切围绕「切换不卸载」

要点：ChatPanel 是面板骨架，MessageList 是消息渲染核心；两层都用了「保持挂载、只切可见性」的策略，用内存换切换时的零重建。

  • workspace 级：App.tsx:357-372 为每个打开的 workspace 挂一个 ChatPanel，非活动的用 invisible + inert 隐藏而不卸载——React 状态、store 订阅、滚动位置全部保留。
  • session 级：ChatPanel.tsx:407-425 为 domCache（LRU，上限 5 个 session，chat-store.ts:375）里的每个 session 各挂一个 MessageList，非活动的 hidden。切 session 不重挂列表，滚动位置不丢。
  • 面板区域：Header（会话名 / 模型名）→ 可选搜索条（⌘F）→ 消息列表 → 右上角悬浮层（WorkflowFloatingPanel + TaskPanel）→ 输入区（ApprovalSurface 与 PromptInput 互斥：审批队列非空时输入框被审批界面替换，ChatPanel.tsx:441-459）→ StatusBar；右侧可挂 DetailDrawer 抽屉。
  • 列表分叉：消息数 ≤ 50 走普通列表（use-stick-to-bottom 吸底），> 50 切 VirtualizedMessageList（MessageList.tsx:38, 105-118）。两者的 renderViewItem 分派逻辑是同一份写法。

2渲染管线：三次分派，每层只关心一种分类

要点：从 store 数组到屏幕，消息经过三层分派。每一层只解决一个问题：第一层把「原始消息数组」整理成「视图项」，第二层按视图项种类选组件，第三层按 part 类型选渲染器。

第一层 · viewItems 前处理（MessageList.tsx:68-76）

const resultMap = useMemo(() => buildResultMap(messages), [messages])          // toolUseId → tool_result 索引
const visibleMessages = useMemo(
  () => messages.filter((m) => !isToolResultOnly(m)), [messages])              // 纯 tool_result 的 user 消息不进视图
const viewItems = useMemo(
  () => pairCliMeta(displayMode === 'result' ? mergeAssistantTurns(visibleMessages) : visibleMessages),
  [visibleMessages, displayMode])
  • buildResultMap（chat-message-adapter.ts:159-174）：把散落的 tool_result 消息按 toolUseId 索引起来，供工具卡片回查——tool_result 从不作为自己的一条消息渲染，它「回到」发起它的工具卡片里（ChatMessageRenderer.tsx:332-368）。这是 Anthropic 转录结构（result 是独立 user 消息）与聊天 UI（结果贴着调用）之间的桥。
  • result 显示模式下 mergeAssistantTurns（message-grouping.ts:81-126）把连续 assistant 消息合并成一个逻辑回合；groupMessageParts 再把回合切成文本区和「过程区」（thinking + tool_use 的连续段），过程区折叠成一行 ProcessRegionGhost，点击开抽屉。
  • pairCliMeta（cli-meta.ts:100+）用正则识别 user 消息里的 CLI 外壳（<command-message>、<local-command-stdout>、<system-reminder>），改判为 meta 视图项。

第二层 · renderViewItem（MessageList.tsx:161-233）

  • meta → SlashCommandMessage 或 MutedSystemNote（灰色系统小注，不是聊天气泡）。
  • isCompactBoundary 的 system 消息 → CompactBoundary 分界线。
  • 其余 → adaptChatMessage（带 WeakMap 缓存）→ ChatMessageRenderer。
注意：VirtualizedMessageList.tsx:405-477 有一份与 MessageList.tsx:161-233 几乎逐行相同的 renderViewItem——两层分派逻辑靠「复制粘贴」保持同步，是这个文件里最明显的重复。

第三层 · part 级渲染（ChatMessageRenderer.tsx）

  • text：user → LinkifiedText（纯文本 + 链接化）；assistant → 先试 detectStructuredReport，否则 CompactableText → Response → Streamdown（流式 markdown，response.tsx:16-28）。
  • thinking → Reasoning 折叠面板。
  • tool_use（:289-373）：Agent → SubagentBriefStatus；Workflow → WorkflowToolCard；TaskCreate/TaskUpdate → 不渲染（它们的数据喂给 TaskPanel）；其余 → 通用 Tool 卡片。
  • 通用 Tool 卡片的输入区查工具渲染注册表（tool-renderers/registry.ts:5-17，Map 结构，index.ts 副作用注册约 30 个：Bash/Edit/Read/AskUserQuestion/ExitPlanMode/TodoWrite…），命中用专用渲染器，未命中 CodeBlock 展示格式化 JSON 兜底（tool.tsx:211-238）。

3历史消息：WS「拉」取，数据源是 SDK 的 JSONL 转录

要点：历史消息不是 REST 拉的，也不是从应用 SQLite 读的——它走 WebSocket 请求/响应，真正的数据源是 Claude Agent SDK 写在磁盘上的 JSONL 会话转录。SQLite（sqlite-store.ts）只存 workspace/session 元数据，不存消息正文。

  1. 触发 — ChatPanel.tsx:158-162：活动 session 没有缓存消息时调用 loadMessages；若正在流式或有缓存则跳过，避免覆盖进行中的状态（chat-store.ts:2786-2810）。
  2. 请求 — wsClient.request('loadMessages', { workspaceId, sessionId })（chat-store.ts:2784-2803），WS 请求/响应模式（带关联 id 的 Promise）。
  3. 服务端 — handleLoadMessages（websocket/server.ts:209-218）→ chatService.loadMessages（chat-service.ts:557-604）→ sdkClient.getSessionMessages（sdk-client.ts:81-86，对 SDK 函数的薄封装，按 sessionId + workspace 目录读 JSONL 转录）。
  4. 归一化 — normalizeSessionMessage（message-normalizer.ts:178-204）把 Anthropic content blocks（text / tool_use / tool_result / thinking）映射为 MessagePart[]，所有 part 直接置 state: 'complete'；压缩边界转成带 isCompactBoundary 标记的合成 system 消息。
  5. 入列 — 客户端 sanitizeMessages（chat-store.ts:422-450）校验整形后写入 messages[sessionId]，窗口裁剪到 200 条；subagents / tasks / workflows 一并水合。
const sdkMessages = await this.sdkClient.getSessionMessages(sessionId, options);
// ...
sdkMessages.forEach((msg: SessionMessage, index: number) => {
  const chatMessage = normalizeSessionMessage(msg);   // ① SDK SessionMessage → 应用 ChatMessage
  if (chatMessage) {
    // ② SDK 不提供逐条时间戳 —— 按索引合成递减时间，只保证相对顺序
    chatMessage.timestamp = Date.now() - (sdkMessages.length - index) * 1000;
    normalized.push(chatMessage);
  }
});

渲染：与实时完全相同。历史落入同一个 messages[sessionId]，走第 2 节的同一管线；tool_use ↔ tool_result 配对由 buildResultMap 在渲染期重建。唯一的历史专属痕迹就是合成时间戳和「所有 part 都是 complete」。

更早历史的分页

虚拟列表滚动到距顶 < 500px 时（VirtualizedMessageList.tsx:266-290，SCROLL_TOP_TRIGGER_THRESHOLD），按 totalMessageCount 算出 offset 调 fetchOlderMessages（chat-store.ts:3139-3181）——同一个 loadMessages 请求带 {offset, limit}（每页 50），结果前插并按消息 id 去重；滚动位置以「首个可见项的 key」为锚在插入后还原（:161-200），用户无感知。

边界情况：① 历史时间戳全部是合成的（SDK 不暴露），不要把消息时间当精确事实用；② bot 会话（WeCom/Feishu）另有 loadMessagesAfter 增量变体（chat-service.ts:606-644 → chat-store.ts:3183），只拉某条之后的新消息；③ 流式进行中不会触发 loadMessages。

4实时消息：WS「推」送，与 2026-06-01 的「整回合」转折

要点：实时通道是为 token 级流式 设计的事件管道——但当前被刻意运行在整回合（whole-turn）模式。两种模式共用同一套事件词汇和同一个客户端累积器，所以切换时客户端一行代码都没改。这是整个 panel 里最值得学的设计。

  1. SDK 流启动 — SessionRuntime.start() 以 PushableIterator 为输入创建 streaming query（session-runtime.ts:227-233）；runMessageLoop 把每条 SDKMessage 喂给 emitter.handle()（:235-243）。
  2. 事件翻译（双模式） — stream_event → handleStreamEvent（sse-emitter.ts:524+，把 Anthropic 的 content_block_start/delta 翻成 text_delta / tool_input_delta / thinking_delta；当前休眠）；整回合 assistant 帧 → handleAssistant（:685-716）→ emitDedupRecovery 逐块补发（:718-777；当前主路径）。
  3. 落环 + 广播 — emitter 的 onEvent 把事件压入 500 条环形缓冲（RING_BUFFER_CAP，session-runtime.ts:22），并扇出到所有 bot / web 事件 handler（:187-212）。
  4. WS 推送 — 每个订阅 socket 把事件包成 {type:'event', eventType:'sse', sessionId, eventId, data} 推给客户端（websocket/server.ts:129-139）。订阅时带上 lastEventId 即从环形缓冲回放丢失事件（session-runtime.ts:484-516）——断线重连不掉字；subscription_ack 携带 serverNonce，发送动作被它 gate。
  5. 客户端累积 — wsClient 单例分发，store 的 handleSseEvent（约 40 个 case，chat-store.ts:1144+）把事件不可变地累积进 messages[sessionId]：assistant_start 建空消息（isStreaming: true）→ text_delta 拼接 → tool_use_start/done 管理工具 part → tool_result 作为独立 user 消息入列（镜像转录结构）→ assistant_done 翻标记。
  6. 渲染 — 没有独立的「流式组件」：Zustand 更新触发同一棵组件树重渲染，memo 比较器保证只有变化的消息真正重绘；Streamdown 负责把（可能未闭合的）markdown 安全渲染出来。

客户端累积器：不可变更新 + 引用隔离

function updateAssistantPart(
  state: ChatState, sessionId: string, messageId: string,
  partIndex: number,
  produce: (existing: MessagePart | undefined) => MessagePart,
): Partial<ChatState> {
  const msgs = state.messages[sessionId] || []
  const idx = msgs.findIndex((m) => m.id === messageId)
  if (idx < 0) return {}
  const parts = [...msgs[idx].parts]
  while (parts.length < partIndex) {           // ① partIndex 之前的缺位补空 text part
    parts.push({ type: 'text', text: '' })
  }
  parts[partIndex] = produce(parts[partIndex])  // ② text_delta 的 produce = 字符串拼接
  const nextMsgs = [...msgs.slice(0, idx), { ...msgs[idx], parts }, ...msgs.slice(idx + 1)]
  return { messages: { ...state.messages, [sessionId]: nextMsgs } }  // ③ 只有这一条消息换了引用
}

③ 是流式性能的关键配合点：每次更新只有目标消息产生新引用，ChatMessageRenderer 的自定义 memo 比较器（:425-468）让其余几百条消息直接跳过重渲染。

关键转折 —— commit 727a41d（2026-06-01）：includePartialMessages: false

SDK 的 includePartialMessages 控制是否发出 stream_event（即 SDKPartialAssistantMessage，sdk.d.ts:4027）。它曾长期为 true；2026-06-01 被改成 false（两处：sdk-client.ts:60、chat-service.ts:1350），commit 信息原文：消除每个 content_block_delta 带来的高频 setState 与 React 重渲染——大文件写入、长回复时的 CPU 尖峰随之消失；代价是「打字机」动画变成整块出现（chunk-at-once），功能行为不变。

为什么客户端零改动？整回合 assistant 帧到达时，handleAssistant 对每个没经流式见过的块调用 emitDedupRecovery，补发同一套事件：

content.forEach((block, index) => {
  if (this.seenStreamPartIndexes.has(index)) {      // ① 增量模式已流过的块：只补 done 事件
    this.closeStreamedBlock(block, index, metaArray?.[index]);
    return;
  }
  this.emitDedupRecovery(block, this.nextPartIndex++, metaArray?.[index]); // ② 整回合帧：逐块补发
});
// emitDedupRecovery 中 text 块的补发（sse-emitter.ts:726-731）：
this.send({
  type: 'text_delta',                               // ③ 同一个 text_delta 事件
  messageId: this.currentMessageId,
  partIndex: index,
  text,                                             // ④ 区别只在：一次性携带完整文本
});

客户端的 text_delta case 只是「往 parts[partIndex] 上拼字符串」——拼一个 token 和拼一整段文本没有任何区别。工具调用同理：增量模式下 tool_use_start → 若干 tool_input_delta → tool_use_done 跨数百毫秒到达；整回合模式下 start 和 done 背靠背到达，inputJsonStream 累积器直接闲置。事件词汇不变，两种模式对客户端透明。

还有个细节：assistant_done 不在 handleAssistant 里发——SDK 会把一个 Anthropic API 消息拆成多个同 id 的整回合帧（thinking 帧、text 帧…），逐帧 finalize 会提前封口；正确时机是 message_stop（stream_event，当前休眠）或 result 兜底（sse-emitter.ts:709-715 的注释）。所以当前模式下 assistant_done 实际由 result 帧触发。

「流式结束」= 就地翻标记，不重新拉取

case 'assistant_done': {
  set((state) => ({
    messages: {
      ...state.messages,
      [sessionId]: (state.messages[sessionId] || []).map((m) =>
        m.id === messageId ? { ...m, isStreaming: false } : m),   // 只翻一个布尔值
    },
  }))
  return
}

从此这条消息与 loadMessages 拉回来的历史不可区分——这正是「sanitizeMessages 与 normalizer 产出同一形状」的意义。磁盘上的 SDK JSONL 转录仍是下次加载时的唯一事实源。

休眠代码清单（设计给 token 模式、当前不会触发）：StreamingToolInputPreview（部分 JSON 实时预览）、Reasoning 的 streaming shimmer、Tool 卡片的 input-streaming 徽标、tool_input_delta 累积。把 includePartialMessages 翻回 true，整条 token 级链路原样复活。

发送路径（反向）

sendMessage（chat-store.ts:2878-2963）：乐观追加 user 消息并把 session 置为 streaming → 有审批挂起则进 draftQueue → 还没拿到 serverNonce 则进 pendingSend（由 subscription_ack 排空）→ 否则 wsClient.request('sendMessage')；服务端 pushMessage 把 SDKUserMessage 推进 query 的输入迭代器（chat-service.ts:1036-1055 → session-runtime.ts:751-760）。

5性能工程：让长会话不卡的五件事

  • 虚拟化：> 50 条切 @tanstack/react-virtual，动态测高、overscan 5、稳定 key（VirtualizedMessageList.tsx:101-112）。
  • 适配缓存：adaptChatMessage 与 mergeAssistantTurns 都用 WeakMap 缓存（chat-message-adapter.ts:46-107、message-grouping.ts:59-67）——引用没变的老消息零成本跳过重算。
  • memo 比较器：ChatMessageRenderer 只在「消息引用 / 自身搜索匹配 / 自身 tool result」变化时重渲（ChatMessageRenderer.tsx:425-468）。这曾是 token 级流式不卡的原因，也正是 727a41d 关闭它能省 CPU 的对价。
  • 窗口上限：DEFAULT_WINDOW_CAP = 200（chat-store.ts:374），pruneWindow 裁最老的消息但绝不拆散 tool_use / tool_result 对（:717-750）——拆了工具卡片就查不到结果了。
  • 吸底两套：普通列表用 use-stick-to-bottom；虚拟列表手动维护 isAtBottom（阈值 50px），新消息滚到末项，并用 useLayoutEffect 观测 scrollHeight 增长保持钉底（:232-254）——因为 result 模式下增长发生在最后一项内部，条目数不变，纯按条数监听会漏。

6要点回顾

  • 一个数组：历史与实时共用 messages[sessionId] 和同一条渲染管线；「流式」只是消息上的 isStreaming 与 part 上的 state 标记，完成 = 翻标记，不重新拉取。
  • 拉推皆 WS：历史走 WS request/response（loadMessages），实时走 WS 推送；SSE-over-HTTP 已是死代码，SseEmitter / eventType:'sse' 的命名是历史遗留。
  • 消息正文在 SDK JSONL 转录里，不在 SQLite；SQLite 只存元数据；历史时间戳是合成的。
  • tool_result 是独立 user 消息（对齐 Anthropic 转录结构），渲染期由 buildResultMap 重新配对回工具卡片；纯 result 消息被 isToolResultOnly 滤出视图。
  • 整回合模式：2026-06-01 起 includePartialMessages: false 关闭 token 级增量（CPU 优化）；整回合帧经 emitDedupRecovery 复用同一套事件词汇，客户端透明；翻回 true 即恢复打字机。
  • 三层分派：viewItems 前处理 → renderViewItem 按 kind → part 级按 type；每层只管一种分类，新消息类型从对应层切入。
