Grounding Context / 背景依据

以下来自代码库扫描、团队既往学习（docs/solutions/）、外部研究（2025–2026 agentic browsing 领域）与五份按轴收集的代码证据档案。

Codebase Context（Comate 现状）

  • 工具面：chat-service.ts 的 buildSdkOptions() 合并两类 MCP server（workspace 级 stdio {name, command, args} + 插件 .mcp.json）；全仓零 createSdkMcpServer。外挂 MCP 工具在 bot 策略里落入 'unknown' 直通 allow-all —— 文档化的 R10 审计缺口，设置页 i18n 已写明 "This is a known audit gap"。
  • 审批面：canUseTool 是唯一 choke point；GUI 有 auto/readonly/manual 三级模式（session-runtime.ts:290-339），AskUserQuestion 的硬门位于 auto 检查之前（"任何模式都必须问人"的先例）；pending_approval 支持断线重发；ApprovalSurface 的 Allow / Allow always / Deny + suggestions 持久化管线已存在；WeCom 模板批准卡复用同一通道。
  • 流式面：单 WS 多路复用，WsEventMessage.eventType 是开放字符串；git-changes 已示范第二条带外订阅通道；500 事件 ring buffer + lastEventId 续播。客户端 RightPanel 的 ContentTab = FileTab | DiffTab 封闭联合类型是设计好的扩展缝；全仓无动态图像负载渲染管线（无 media_type 处理）。
  • 打包含：esbuild → pkg sidecar 管线；Claude binary / ripgrep / better-sqlite3 经 externalBin + resources 分发；resolve-sdk-binary 四级策略梯被 ripgrep 显式复用为模板（"Mirrors resolveSdkBinary's strategy ladder"）；CSP 已放行 http://localhost:* 与 ws://localhost:*；签名自动更新管线使每 MB 捆绑成为逐平台 recurring 成本。
  • 数据面：workspace settings 是扁平可选字段袋（browserPermissions 零迁移落点）；workspaces 表已有 skills/mcpServers/hooks JSON 列；全仓 cookie/浏览器足迹为零（绿地）；无可重复工作流实体；wecom-file-storage 的工作区落盘校验与 wecom-queue-worker 持久队列是现成模板。

Past Learnings（docs/solutions/）

  • 与打包的 minified SDK 内部对齐必须 byte-for-byte，并用硬编码测试向量做漂移警报（2026-07-13）。
  • 团队的 recurring wound 是 silent-empty success —— 浏览器提取/动作必须"响亮失败"（存在性探针 + 显式错误事件）。
  • 长生命周期流的纪律：15s 服务端心跳 + 客户端读超时 >2x、keepalive 不进 ring buffer、稳定 eventId 重放锚点、身份守卫退订、重连重发 pending approvals。浏览器自动化正是"长连接上的长空闲段"，这些教训全部适用。
  • 签名自动更新管线与 tauri.conf.json + CI 紧耦合；native binary 捆绑有先例，但浏览器运行时比现有全部资源大一个数量级。

External Context（研究价值：high）

  • 控制面三种架构：vision 循环（Anthropic Computer Use，截图进出、token 贵）；a11y/DOM 蒸馏（Playwright MCP ~22 工具、稳定元素 ref；browser-use 89.1% WebVoyager）；混合（Stagehand 的 act/extract/observe + v3 动作缓存，重复工作流近零成本，"replay determinism"）。
  • 嵌入路径硬约束：Tauri 系统 webview 只有 Windows WebView2 暴露 CDP；macOS WKWebView / Linux WebKitGTK 均没有 → sidecar 内 headless Chromium + 画面流回 React 是与 Comate 架构匹配的路径；Steel.dev 证明 localhost 浏览器服务 + 会话查看器是可行的产品形态。
  • 安全共识：提示注入共识未解 —— OpenAI 称 "unlikely to ever be fully solved"；Claude for Chrome 红队数据 23.6%→11.2%（Simon Willison 称缓解后仍是 "catastrophic failure rate"）。新兴防御栈 = 站点级权限 + 逐动作确认 + 登出模式 + 架构隔离 —— Claude Code 式权限系统是天然载体。
  • 生态空位：Comet / Dia / Atlas 都是独立浏览器、不可嵌入（The Register）；"local-first AI workspace 内嵌 agentic 浏览器"无人占据；Tauri 的 WKWebView CDP 缺口意味着多数 Tauri 应用做不了这件事。差异化轴 = 你授予多少自主权。

可视化浏览器面板：ghost-tab 架构 + 双向人机接管

preview-observation-ux Confidence 85% Complexity High
Description Steel.dev 式的本地浏览器服务 + 内嵌实时面板，这是整个功能的产品中心。无头 Chromium 跑在 sidecar（运行时来源见 #5），RightPanel 新增 BrowserTab（ContentTab = FileTab | DiffTab 联合类型的扩展点）显示实时画面：CDP screencast 帧经 WS 第二订阅通道流入（git-changes 先例，帧刻意不进 500 事件 ring buffer），面板内键鼠经 CDP Input.dispatch* 回传浏览器。双向人机接力是一等交互：agent 撞墙（登录墙、CAPTCHA、二维码、歧义表单）时发出 browser_handoff 待决事件 —— 完全复用 pending_approval 的注册、SSE 发射、断线重发、POST 决议路由；用户在面板中直接操作（输用户名密码、扫码、完成一段复杂流程），然后点"继续"，agent 带着状态差分（URL 变化、DOM 增量、表单已填值 —— 密码类字段脱敏）接手后续流程。用户也可以随时主动接管 —— 面板背后是 agent | user | paused 的独占输入状态机，任何时刻只有一个控制者。聊天流里的语义动作日志（带 eventId、可续播、15s 心跳）作为互补叙事，截图按需拉取。凭据永不入模型上下文 —— 用户输密码时模型只看到"已登录"这一事实。
Basis direct: 用户明确的产品设想 —— "Steel.dev 类似的，在 comate 中有一个内嵌的 panel，让用户可以看到 ai agent 在怎么样操作浏览器，需要人介入的时候，人可以直接在 panel 中操作，如输入用户名和密码，扫二维码等，或者用户在网页中进行一些复杂的操作之后，通知 agent 继续后续的流程"。技术锚点：session-runtime.ts:807-809 pending approval 注册 + :458-459 断线重发；chat.ts:209-244 审批决议路由；websocket/types.ts:41-48 开放 eventType 字符串 + git-changes-service.ts:115-128 第二订阅通道；tauri.conf.json:26 CSP 已放行 localhost。 external: Steel.dev 的 stateful sessions + 会话查看器形态；航空双人制操纵权交接（控制权威模糊是同构失效模式）；瘦客户端游戏串流（帧下行、输入上行、接管成熟模式）；Tauri WKWebView 无 CDP → 画面必须来自 sidecar 浏览器而非应用 webview。
Rationale 不能看见、不能接管的"受控"只是营销词汇。面板把"agent 在干什么"从信任问题变成观察问题；双向接力把登录墙、二维码、复杂流程这些 agent 死区变成"人做三秒、agent 做三分钟"。WKWebView 的 CDP 缺口意味着多数 Tauri 应用做不出这个形态 —— 这是差异化本体，也顺带成为凭据不入上下文的登录通道（与 #4 互为因果）。
Downsides 像素流是心跳/重连/带宽的三重债务（已被产品决策接受为核心成本 —— 可靠性纪律必须照搬团队 SSE 学习：15s 心跳、读超时 >2x、keepalive 与帧不进 ring buffer）；帧率与延迟需要调优；DOM 状态差分与字段脱敏实现复杂；用户不在场时 handoff 会阻塞任务（需超时与降级策略）；面板键鼠回传承载了真实安全语义。

一等自研浏览器工具面（含零-token 页面蒸馏）

agent-tool-surface Confidence 85% Complexity Medium
Description 在 sidecar 进程内托管 Playwright，以稳定工具名暴露一小组意图级工具（BrowserOpen / BrowserSnapshot / BrowserAct / BrowserSubmit / BrowserExtract），走 a11y 树 + 稳定元素 ref，模型循环里不放截图。同时内置服务端蒸馏器：每个页面压缩成"页面模型"（表单及命名字段、可点元素及 ref、正文抽取，合计数百 token），原始 DOM 在任何路径上都不进上下文 —— 蒸馏器既是 token 经济的根，也是提示注入的第一道防火墙（页面原文永远只是数据）。自有工具名是稳定契约：权限分类（新 browser 类别）、tool-renderer、审批 UI、工作流录制都能按名精确挂接，避开外挂 playwright-mcp 的 ~22 个 schema 吃上下文、以及外部 MCP 工具落入 R10 'unknown' 直通 allow-all 的缺口。传输层可以用 SDK 的 createSdkMcpServer（handler 跑在 sidecar 进程内，直调 browser-service，零 IPC）—— 拒绝的不是 MCP，而是第三方 MCP 的 schema 所有权。竞争路线需在评审中比较：CLI 插件（继承 Bash 门控，但有状态浏览器如何穿越单次 CLI 调用的问题未解）与脚本即动作格式（其精华活在 #6 中）。
Basis direct: chat-service.ts:1451-1453 — "'unknown' = tool not in any category (MCP, Skill, future SDK built-in / without a category fit). Fall through to today's allow-all behavior per R10."；workspace.ts:41-45 的 McpServer 仅 stdio 三字段；全仓 createSdkMcpServer 零命中（证据档案核实）；tool-renderers/registry.ts:5-16 按工具名的渲染注册表。 external: Playwright MCP README 自警 tool schema 上下文消耗、勿叠两个浏览器 MCP；Computer Use 截图循环 token 成本高（vision 循环 10 字段表单 ≈ 3-5 万 token，蒸馏模型 + ref 动作 ≈ 2-3k，差一个数量级）。
Rationale 工具面形状是所有后续子系统（权限、渲染、录制、审批）的地基选择：工具名归 Comate 所有，六类权限策略才能"看见"浏览器动作；选外挂 stdio MCP，这些全部落空。蒸馏器的失败设计必须"响亮"（存在性探针 + 显式错误事件），回应团队 silent-empty success 的 recurring wound。
Downsides 进程内 Playwright 增加 sidecar 内存与复杂度；自研 schema 需随模型行为持续调优（外挂 MCP 生态迭代免费）；蒸馏器可能丢失页面细节，需定向 DOM 查询兜底；createSdkMcpServer 在打包版 SDK（0.3.207）的确切形状需按 byte-for-byte 学习验证。

站点级权限台账 + 不可逆动作硬门

permissions-safety Confidence 85% Complexity Medium
Description 类比银行收款人登记与分级授权：WorkspaceSettings 新增 browserPermissions（扁平字段袋零迁移），按 域名 × 动作类 判决。这把权限模型从"按工具名"升级为"按 URL + 动作语义" —— 是 Comate 第一个按参数判决的权限评估器。配套两层执行：CDP 请求拦截层的 origin 白名单硬边界（方案外导航物理失败，bot 会话可因此安全开启浏览器），以及审批疲劳的语义层答案 —— 计划预授权（动手前产出结构化行动计划，用户一次批准整份方案，browser-service 运行时强制方案包络）。提示注入的信任边界由 #2 的蒸馏器承载（内容永远是数据），敏感页面（密码框/支付/checkout）再叠加阶段级工具面收缩。
Basis direct: session-runtime.ts:306-314 — AskUserQuestion 特判位于 :317 的 auto 检查之前，即"无论何种模式都必须问人"的硬门先例；:331-338 的 suggestions?: PermissionUpdate[] 透传与 ApprovalSurface.tsx:327-356 的 "Allow always" 构成持久化授权端到端管线；workspace.ts:18-25 settings 字段袋先例；tool-permission-policy.ts:39-45 的 network 类别与 bot-path-policy.ts:20-29 的 deny-glob（文件侧 deny → URL 侧 allow-origin 的同构迁移）。 external: 银行收款人/限额模型（用户已理解二十年的心智模型）；Claude for Chrome 站点级权限 + 动作确认（红队 23.6%→11.2%）。
Rationale 用户点名要"提交" —— 提交恰恰是不可逆动作，必须成为独立于工具名的动作类。台账随工作区沉淀、越用越安静；评估器一旦存在，未来任何按参数分级的工具都有模板。
Downsides 参数级判决引擎是第一个先例，设计错误会波及所有未来工具；域名/页面分类启发式可能误判（子域、CDN、iframe 嵌套）；审批 UI 复杂度显著增加；台账与计划预授权、环境档位三套机制需要统一的优先级语义，否则用户困惑。

登录态托管：默认登出 + 接管式登录 + storageState 钥匙托管

extraction-sessions-workflows Confidence 75% Complexity Medium
Description 类比银行保险箱钥匙托管：登录一律走 #1 的接管通道 —— 用户在面板里直接输密码、扫码，模型只见"已登录"这一事实；Playwright storageState 按 (workspace, 域组) 加密存于 COMATE_DATA_DIR —— 绝不落 folderPath、绝不进日志、禁止跨工作区导出；每次 agent 使用留审计，"退出登录" = 文件销毁。bot 会话强制 ephemeral（一次性登出 profile，任务结束即焚）或按 data/<user> 用户目录隔离，延续 sanitizeBotEnv 的凭据剥离哲学。默认登出同时是注入缓解（未登录页面可破坏面小得多），并让读取类动作可进 GUI auto/readonly 的免审批集合 —— 自主权拨盘从"审批频率"挪到"环境隔离档位"。
Basis direct: 证据档案核实 —— 全仓 cookie 零命中（纯绿地，无遗留约束）；data-dir.ts:12-13 的 COMATE_DATA_DIR 解析；bot-path-policy.ts:47-48 的 per-user 目录隔离惯例；chat-service.ts:37-46 sanitizeBotEnv 按会话类别剥离凭据先例；CLAUDE.md "never log secrets"。 external: 托管钥匙管理（登录态是 bearer credential，托管纪律是已知解）；Steel.dev 的 stateful /sessions 是核心卖点；Atlas logged-in/out toggle + Lockdown Mode；Dia local-first 加密存储。
Rationale "通过 prompt 操作网页"的真实价值大半在登录墙之后：没有托管，每个任务都重新登录（体验死亡）；有托管无纪律则是凭据泄露温床（安全死亡）—— 托管模型是两者唯一的同时解，也是与 Chrome/Atlas 差异化（本地、按工作区隔离、默认登出）的定位支点。
Downsides 加密密钥管理未解决（系统 keychain？主密码？Tauri 侧安全存储？）；storageState 过期/被服务端吊销时的体验需要设计；这是 Comate 第一个"凭证相邻"资产类别，企业合规与审计要求会随之而来。

运行时零打包：解析注册表 + 本机 Chrome 优先 + 懒下载兜底

browser-runtime-bundling Confidence 85% Complexity Low
Description 不把 Chromium 打进安装包。把 resolve-sdk-binary 的四级策略梯泛化为"可选运行时注册表"：系统 Chrome/Edge（--remote-debugging-port 受控启动，Comate 专属隔离 profile）→ 工作区配置路径 → 首次使用时懒下载 headless Chromium 到 COMATE_DATA_DIR（下载进度走 WS 事件）；配 /api/health/browser 的 resolve-then---version 探测，缺失时发显式错误事件而非静默失败。安装包零增量，签名自动更新管线不背每平台 ~150MB 的 recurring 成本；macOS universal 双架构问题随之消失。第二阶段可加双运行时信任拓扑：洁净室（每任务全新 storageState，bot/不可信默认）与出诊（CDP 连接用户 Chrome，继承登录态），策略层按会话信任姿态选择。"下载-即-能力"的基础设施未来可被任何重型可选运行时复用。
Basis direct: file-search.ts:44-49 — "Mirrors resolveSdkBinary's strategy ladder so the same call works in dev, pkg-bundled sidecar, and Tauri production"（梯子被显式写成可复用模板）；index.ts:139-151 的 resolve-then---version 健康探测模式；build-sidecar.ts:156 的 macOS 双架构构建。 external: Playwright 已是 devDependency，playwright install chromium 即懒下载先例；Steel.dev 证明 localhost 浏览器服务中间路径。 reasoned: 学习文档指出签名自动更新管线与每 MB 捆绑成本紧耦合、逐平台递归计费；懒下载把 recurring 成本变为一次性首用成本，且"不用者不付费"。
Rationale 打包 vs 解析 vs 下载是整个功能的第一个岔路口，选错方向，安装包从几十 MB 涨到数百 MB，三平台签名更新管线每次发版都为此付利息。这是成本结构决策，必须动工前拍板。
Downsides 懒下载需要下载基础设施（进度、断点续传、校验、可能的镜像源）；本机 Chrome 版本漂移带来兼容性测试面；企业受管环境可能无 Chrome 或禁止调试端口；首次使用时的下载等待是 onboarding 摩擦点。

录制-回放工作流：审批-执行-沉淀一条龙

extraction-sessions-workflows Confidence 65% Complexity High
Description 三个独立候选咬合为一条流水线：agent 产出结构化计划 → 计划编译为可审查 Playwright 脚本（审批卡用既有 tool-renderer 渲染成代码块，"批准计划一次"）→ 执行成功的脚本一键冻结为参数化工作流（a11y ref 解析为自愈选择器包 + 提取 schema + NL 触发描述，落 workspaces.skills JSON 列先例或新表）。之后同类任务确定性回放、零模型 token，模型只在选择器漂移时介入自愈；可从聊天、定时、WeCom bot（proactive queue）三个面触发。"盯住这个信息源"的物化视图（browser_watches：schedule + hash diff + 队列推回会话）作为其定时特化。回放不把页面内容进上下文，注入暴露面同步缩小。站点领航员（按域名注入的站点知识文件，含稳定选择器）是后续可加的知识层，让回放更稳。
Basis direct: sqlite-store.ts:76-79 — skills/mcpServers/hooks JSON 列的 per-workspace 能力包先例；证据档案 §8 — "No persisted entity for reusable workflows/pipelines"（最近的是 workspace_prompt_history）；wecom-queue-worker.ts:7-11 持久队列 + wecom-bot-service.ts:1148-1162 按用户复用会话。 external: Stagehand v3 action caching 让重复工作流近零成本（"replay determinism"）；RPA/UiPath 二十年教训 —— 语义自愈选择器是脆选择器问题的解；Dia 的用户自定义 NL Skills。
Rationale 没有回放，浏览器功能的经济模型是"每次都全价" —— token 贵、慢、还会随机失败。录制-回放把"agent 一次性成功"转化为"资产性自动化"：第一次付全额模型成本，之后每次近零。这是从聊天功能跃迁为自动化产品的分水岭，也是唯一随使用次数增值的资产，直接构成留存理由。
Downsides 体量最大：录制器 + 回放引擎 + 自愈 + 新实体 + 三触发面；自愈选择器仍会漂移失败（需要失败通知与人工修复路径）；错误回放的破坏力（自动提交错表单）要求回放同样过提交硬门（#3）；"v1 就建护城河"还是"先验证单步体验"是真实的排序争议。
